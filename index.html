模型调试记录

1.base_base_o


不使用slim策略训练，不使用slim策略导出的最普通模型，耗时是3500ms。

2.base_base_set_o


在1的条件下追加了PaddleLite的Opt工具进行8bit优化，耗时几乎没变化。

3.base_qat_o


不使用Slim策略训练，使用8bit量化导出模型，耗时减少了一半。
没有正确识别目标。

4.base_qat_set_o


在3的条件下追加opt，耗时几乎没变化。
没有正确识别目标。

5.qat_base_o


使用8bitQAT进行训练，但是不使用Slim策略导出模型，耗时回到了3500ms。
没有正确识别目标。

6.qat_base_set_o


没什么可说的，同上。

7.qat_qat_o


使用8bitQAT量化训练，使用8bit量化导出模型，耗时同3比未见减少。

8.qat_qat_set_o


同上

结论

1.只有在训练时的Slim策略与模型导出时的Slim策略相同时，得到的模型才可以得到正确的结果。
2.无论是在模型导出环节还是在Opt环节中加入8bit量化，都会使模型变小，但模型的大小并不能决定预测的速度。
3.被我们寄予厚望的Opt工具效果甚微，甚至是没有效果，也有可能是使用方法出了问题。

发表于 2022-04-17 更新于 2022-04-19 分类于 FPGA

问题概述

在使用PaddleDetection训练模型修改训练图片输入大小时遇到了如下的错误

ValueError: (InvalidArgument) The 2-th dimension of input[0] and input[1] is expected to be equal.But received input[0]’s shape = [1, 256, 20, 20], input[1]’s shape = [1, 512, 19, 19].
[Hint: Expected inputs_dims[0][j] == inputs_dims[i][j], but received inputs_dims[0][j]:20 != inputs_dims[i][j]:19.] (at ..\paddle/fluid/operators/concat_op.h:63)
[operator < concat > error]
大致意思时输入的参数与预期不符等等（我还没有完全搞清楚）。

起因：修改config文件

首先来看一下在发生这个错误前我对yolov3_reader.yml做的修改：
我将batch_transforms中初始的- BatchRandomResize: {target_size: [320, 352, 384, 416, 448, 480, 512, 544, 576, 608], random_size: True, random_interp: True, keep_ratio: False}
修改为- BatchRandomResize: {target_size: [160, 176, 192, 208, 224, 240, 256, 272, 288, 304], random_size: True, random_interp: True, keep_ratio: False}，本意是Resize训练输入图片，让模型变小一些，然后就出现了上述报错。

排查原因

首先根据报错的字面意思，我怀疑问题与输入图片的shape有关。
由于默认是开启了random_size的，因此每次读一个batch的训练图片时会将图片Resize成target_size列表中随机的一个大小，我推测报错的原因与target_size有关。我尝试将列表中的其它数值删除，只留下256、320等单个数值，发现个别数值可用。反复测试后发现32的整数倍的size均可用，例如target_size: [288, 320, 352]。

解决方案

可以通过修改BatchRandomResize中的target_size来达到修改输入图片size的目的，但注意size要是32的倍数，即可避免该报错。同样的，如果对它进行了修改，那么在下面的EvalReader和TestReader中也要对target_size做相应的修改，否则在评估阶段也会出现同样的报错。

一点思考

实际上我们可以看到，在初始的BatchRandomResize中，给出的target_size实际上都是32的倍数，所以理论上导致该报错的原因应该是PaddleDetection框架中做的限制，这一点在源码中应该有体现，欢迎讨论！

飞桨(PaddlePaddle)是集深度学习核心框架、工具组件和服务平台为一体的技术先进、功能完备的开源深度学习平台，已被中国企业广泛使用，深度契合企业应用需求，拥有活跃的开发者社区生态。提供丰富的官方支持模型集合，并推出全类型的高性能部署和集成方案供开发者使用。

安装PaddlePaddle环境

我使用了Anaconda去配置Paddle环境。
使用Anaconda Prompt为Paddle创建一个新的环境，在Anaconda命令行中输入下面的代码，即可安装PaddlePaddle2.2，CUDA版本为10.2。我的显卡是支持CUDA11.2的，但我在安装CUDA11.2的PaddlePaddle时出现了一些错误，我换装了CUDA10.2的PaddlePaddle后解决了这个问题，并且使用上一切正常。

安装完成后找一个可以执行python代码的软件，我使用的是Pycharm（理论上仅用命令行也可以，在命令前加上python表示用python执行，未经验证），选定之前创建的Paddle环境后输入import paddle ，再输入 paddle.utils.run_check()，如果出现PaddlePaddle is installed successfully!，说明已成功安装。

安装PaddleDetection

在PaddlePaddle环境的根目录下新建一个文件夹用于安装PaddleDetecetion，在Anacon Powershell中cd到该目录。

# 克隆PaddleDetection仓库

git clone https://github.com/PaddlePaddle/PaddleDetection.git

# 安装其他依赖
cd PaddleDetection
pip install -r requirements.txt

# 编译安装paddledet
python setup.py install

我的电赛经历

发表于 2022-02-21 更新于 2022-03-30 分类于 咸盐碎雨

@TOC

听歌

lemon tree

前言

对于每一个电子人来说电赛都会是一段难忘的经历，四天三夜高强度比赛带来的感受只有经历过的人才会懂。我们队在竞赛过程中遇到了不少波折，最后也只是堪堪拿了个奖，还远远没有达到预期。虽然实在是不甘心承认自己技不如人，但既然已经失败了说什么都是借口，还是老老实实把竞赛过程中的所见、所思、所想老老实实记录下来，牢记教训吧。

竞赛过程

选题

第一天的上午，拿到赛题压缩包后我们就开始了紧张的选题工作。初步分析后，我们认为有三道题是我们可选的，分别是A题：信号失真度测量装置、E题：数字-模拟信号混合传输收发机、F题：智能送药小车。

然后我们开始仔细研究在这三道题中，我们队分别有哪些优势。在A题中，A题本身题目难度不大，我们一看到A题的题干就有了清晰的解题思路，再从系统实现的角度来讲，我们队伍既擅长单片机又擅长FPGA，对于这种信号处理类的题目能占到很大的便宜，让我们顾虑的是题干中有这样一句话：

“本题用于信号失真度测量的主控制器和数据采集器必须使用 TI 公司的MCU 及其片内 ADC，不得使用其他片外 ADC和数据采集模块（卡）成品。”

这句话该如何解读呢？（这是一个大大大大大伏笔）抱着疑问，我们开始研究E题，E题主要内容是通信原理里面的知识，难也没有很难，但不利于发挥我们的优势，属于是比上不足比下有余的题。最后是F题，F题中的数字识别的部分我们在比赛前已经研究过了，整套系统可以高质量地运行，如果我们选择这道题，那么相当于一半的工作已经很好的完成了，问题就在于我们三个人没有调试小车的经验。

分析到这里已经比较明了，无论选A题还是F题，我们都要承担一部分的风险，E题在我们这里优先级不高。A题的风险在于我们担心他们不允许使用FPGA，其实从字面的角度来讲风险非常低：赛方说的是主控制器和数据采集器必须使用TI公司的MCU及其片内ADC，那么除此之外的部分理应让我们自行设计对吧？（不知道在看的你如何理解这句话呢）在我们眼里F题的风险显然要比A题更高，我们有把握调好小车，但对于这个只有一次机会的比赛来说，“有把握”还是远远不够的。思量再三，我们选择了A题。

做题

我们大致分了一下工，我的工作是充分发挥FPGA的优势，对采样得到的信号做FFT，和THD的计算。Xilinx的FFT IP核我以前调试过，但由于年代久远调试时遇到了一些问题，我转念一想，整道题对运算时间的要求并不高，我可以用PS端做FFT，这能省下不少的设计时间。（当时感觉有些投机取巧的味道在里面，但到评审的时候发现人家测评的老师也确实不在乎这种做法）

到了下午的时候突然不知道从哪来了个所谓的“赛题说明”其中说到：

A题主控器和数据采集用TI芯片，数据处理能否用其他MCU？ 答：只能使用TI的MCU芯片。

这个解释让我们难以接受，赛题中明明没有提及数据处理这部分的要求，却在选完题后才给一个补充说明说只能使用TI的MCU芯片，甚至这个补充说明本身也有语义不明的问题：究竟是“数据处理禁止使用其它MCU，只能使用TI的MCU”，还是“整个系统只能使用TI的MCU”?无奈的是比赛还得继续呀。但这也导致我之前用FPGA做的工作很大程度上白费了，我当时几乎没有STM32的经验，没办法做移植，这也导致队友的工作压力也一下子变得很大，甚至能不能完成整个设计都是问题。

我决定还是先按原计划，用FPGA做个成品出来，如果真的来不及完成的话就用这个FPGA顶上（属于是下下策了）。我认为出现这样的问题是因为赛方自己没有考虑到会有学生用FPGA来做数据处理，导致赛题说明出现了漏洞，既然如此我也可以就这点来和他们沟通，毕竟赛题白纸黑字在这，我也是有根有据没有瞎说。

到了第二天的中午，FPGA版的作品的雏形被我赶出来了，已经可以完成赛题中要求的功能，还需要进一步调试来把精度调校到一个理想的水准。但由于FPGA版的作品不能轻易提交，而且强大的队友们在这种情况下有了较大的进展，商量过后我们决定放一放FPGA，把重心放到MSP432的实现上。

接下来两天时间里就是不断的调硬件、写文档、帮助写代码的队友查文献找精度缺失的原因······

好在队友水平确实够高，最终几乎达标了所有指标，但由于我们硬件选型有些问题，测试的最后一组数据又比较苛刻，没能进入国赛。

总结

以上的内容都是在我充满怨念的情况下写出来的，用词难免有点偏激。我觉得这次比赛我真的需要感谢一下我的队友，在这样的情况下还能得奖靠的真的是他们高超的素养，作为队长我有的时候也会觉得我是拖后腿的那一个。

我自己确实也出了问题，客观问题不谈，主观上的错误至少有以下几点：
1.比赛前有些膨胀了，看到题目后跟着脑子里的思路一起上了头，老师说当时我已经隐约察觉到了有出现上述问题的可能性，但我认为概率极小最后选择了忽略。
2.缺少跟老师的沟通，其实当时最稳妥的选择是和老师去一起找主办方对规则进行确认，但我由于怕麻烦忽略了这一点。
3.硬件选型，即使当时我们打算用FPGA做，仍然应该仔细对比各板之间的参数，这点也是导致我们最后达不到最后一个指标的重要原因，这也侧面体现了我的理论知识并不够扎实。

所有失败的原因归根结底还是自己水平有限，即使主办方的规则说明再不合理，如果我的STM32有FPGA一样的水平，我相信我们也一定能进入国赛。

牢记教训！

基于MicroBlaze内核的SOPC系统设计

发表于 2021-12-25 更新于 2022-02-01 分类于 FPGA

把你的FPGA变成单片机——基于MicroBlaze内核的SOPC系统设计

听歌

斯卡布罗集市

## 前言

这是我第一次使用MicroBlaze处理器软核，基于MicroBlaze的可编程片上系统可以根据用户的需要搭建外设，大大增强了系统的灵活性，在手头上没有Zynq的情况下，这或许是一个很好的替代选项。其实主要是一个学习的过程，系统总体设计难度不大，熟悉Block Design和SDK的提供的各种API后很快就能上手。

设计过程

首先添加所需的各模块到Block Design中，配置各模块的功能与参数，随后点击自动连线，让系统自动配置AXI等接口的连接方式，最后引出Uart、GPIO等模块的外部引脚，至此Block Design搭建完成。

右击Block Design选择Generate Output Products，然后点击综合，综合完成后打开综合结果，查询开发板的用户手册配置相应的管脚。之后进行布局布线，最后生成bit流文件。

依次点击File、Export、Export Hardware，导出相关文件，随后选择Launch SDK，至此硬件开发部分结束，进入SDK软件开发环境。

在SDK开发环境中创建一个新的工程。首先要定义各模块的相关参数，包含GPIO、XADC、中断以及IIC等IP核的器件ID、相关寄存器的地址、所需变量和句柄并定义中断所需的回调函数。

Xilinx的7系列FPGA器件在片上集成了模数转换器和相关的片上传感器（例如内置温度传感器和功耗传感器），可以采集模拟信号转为数字信号送给FPGA内部使用，有力地提高了系统的集成度。我们通过它获取芯片结温和FPGA的若干供电电压，用于监控FPGA内部运行状况。

其中需要注意的是XADC模块的采样通道并不是全部可用的，每块开发板都有所不同，需要预先查看开发板的用户手册来确定使用哪个通道。Xilinx在IP核中已经预先为每一个通道分配了对应的地址，使用过程中一定要注意地址的问题，例如AD采样通道7的地址是0x25C，我们可以通过这个地址来访问AD采样通道7的采样值。此处的0x25C是16进制的地址，在SDK中定义的是偏移地址（以字节作为单位），两者有所差别。在使用过程中需要仔细阅读Xilinx提供的相关手册。

IIC模块在运行的过程中每接受/发送成功一个字节后会发起一次中断。我们编写了相关代码，使得主机在完成了发送一个字节的动作后或是正在接收一个字节时，就会进入阻塞状态，随后利用中断回调函数来清除中断标志位使得程序继续运行，这样的好处是可以用非常简单的逻辑来对齐IIC的时序，便于控制。

在主函数中，我们先使能各模块并调用函数初始化相关模块，随后程序进入while循环，设备开始工作。

调试过程

第一个问题是有关GPIO库函数中设置GPIO方向的函数XGpio_SetDataDirection，

我们使用该库函数定义GPIO的方向后发现GPIO功能异常，只有其中的最低位可以正常工作，但是当我们使用配置寄存器的方式去修改GPIO的方向时功能正常。我们查阅了2018.3版本的Vivado SDK官方API手册，其中说到该函数的第三个参数为方向掩码，设置为0时GPIO的方向为输出，设置为1时GPIO的方向为输入。我们的代码满足要求，经过我们的反复确认，手册中提及的这个方向掩码实际上是针对单个GPIO而言的，该版本下的API手册中并没有说清楚当用户使用多路GPIO时该参数需要如何赋值，在查阅最新版本的手册后发现方向掩码的描述应是设置为0 的位为输出，设置为1 的位为输入。也就是说当使用多路GPIO的时候应给方向掩码的每一位赋1而不是单纯的将方向掩码改为1。这与通过修改寄存器来修改GPIO的方向的方法一致，修改过后成功解决问题。

第二个问题也是耗时最久的问题，程序在通过IIC读取MLX90614的温度时卡死。经过仔细排查，最终锁定问题出在回调函数的设定函数（XIic_SetRecvHandler, XIic_SetSendHandler以及XIic_SetStatusHandler）上，查询它们的子函数，发现在一个名为Xil_AssertVoid的函数中包含一层While循环，而这层While循环在程序运行时并没有正常跳出，这是问题产生的原因。查阅资料后发现，这个函数在调试模式下会调用Xil_Assert来检查参数是否正常。如果不正常，缺省情况下，没有打印，会进入死循环。通过调用Xil_AssertSetCallback，设置回调函数，可以自己处理失败情况，比如增加打印。修改While循环的条件后问题解决，程序正常运行。

工程

最终设备能够测量物体温度并识别人员是否靠近、是否佩戴口罩（用的Openmv，神经网络识别）以及体温是否正常，最终实时语言播报并把相关信息发送到手机上。

FPGA_FIR数字低通滤波器

发表于 2021-07-21 更新于 2022-01-28 分类于 FPGA

听歌

梦龙的Demons

# 前言

这些天在准备电赛，看到其它同学有在准备一些模拟滤波器，我想起来Vivado中似乎带有XADC以及FIR滤波器的ip核，只需要再准备一块DA模块来输出信号就可以组成一个数字FIR滤波器。查阅相关资料后感觉难度并不大，但实际开始调试时仍然遇到了许多问题折磨了我很久。网上很多资料都比较笼统，没有对各个信号的用法作说明，本文从一个第一次接触XADC和FIR核的小白视角来介绍一下它们的使用方法以及调试中的一些问题。

一、XADC核的配置以及使用方式

1.滤波的第一步我们需要把外界的模拟量转化为数字量，在这里我使用了Xillinx 7系列FPGA自带的XADC，当然如果你想外接其它AD模块也完全没有问题。在设置IP核时，主要关心以下几个配置。


第一个标签页：

1.选择DRP模式，这个模式相对来说便于控制。
2.在Timing Mode中选择Continuous Mode，其意义是对信号持续进行采样。
3.选择Simultaneous Selection,这个选项可以使得XADC可以对多路信号进行采样，如果被采样的信号只有一路的话选择Single也没有问题。
4.然后设置系统的工作时钟以及采样率，XADC支持的最高采样率为1Mhz。

第二个标签页：

1.Channel Averaging指的是对输入的信号做平均运算，好处是使得输入的信号更加平滑没有毛刺，坏处是大大降低了采样率，会使高频信号失真。例如：输入信号为10KHz，不使用平均运算时每一个周期的信号会被采样1M/10K=100次，此时波形基本不失真，但如果对它做16位的平均那么该信号只相当于被采样了100/16次，显然波形会失真。在该数字滤波器的设计中我们选择None就可以了。

第三个标签页：

1.我只使用了一路信号输入，勾选va0通路即可。

编写XADC驱动：

XADC的驱动非常简单，只需要每隔一段时间拉高读使能端即可，代码放在文章的最后。到这里XADC的部分基本完成了。

基于FPGA的闸门时间为1s的计频器

发表于 2021-05-13 更新于 2022-01-28 分类于 FPGA

@TOC

听歌

The sound of silence—-Simon & Garfunkel
老歌了。

前言

这是学校的电子设计比赛中的一道题，我们队伍用basys3开发板做了这么一个计频器，下面提到的功能基本上都达到了，程序上略微还有一些瑕疵。

功能

  1. 测量正弦波信号的频率，1Hz-10MHz，峰峰值在20mV-1000mV，相对误差误差在10e-2以内；
  2. 测量两个同频不同相位的方波信号的时间间隔，峰峰值在100mV-1000mV，相对误差误差在10e-2以内；
  3. 测量方波信号的占空比，相对误差误差在10e-2以内；

测得的数据稳定，刷新时间不大于2s，自动显示单位。
（其实就是2015年全国电赛计频器的弱化版）

总体思路

1.测量频率：不断检测目标信号，一直到出现上升沿，此时打开1s的闸门，记录1s钟之内出现的上升沿、下降沿个数以及高电平、低电平的持续时间。如果上升沿的个数=下降沿的个数就说明上升沿的个数与高电平的时间是匹配的；如果上升沿的个数>下降沿的个数则说明下降沿的个数与低电平的时间是匹配的；由于是出现上升沿的时候开启闸门，所以理论上不会出现下降沿个数>上升沿个数的情况。

上升沿的个数>下降沿的个数 –> 频率=高电平时间/上升沿个数
上升沿的个数=下降沿的个数–> 频率=低电平时间/下降沿个数

2.测量时间间隔：将两路信号做异或运算，高电平的时长即时间间隔。*==需要注意的是当两路信号的相位差正好是180°时异或运算得到的信号全为高电平，这种情况为妨碍测量，需要特殊情况特殊处理。==*

3.占空比测量:在一个上升沿和一个下降沿之间，记高电平和低电平的时间

流程图

模块功能说明

检测模块：①测量闸门时间内的上升沿个数、下降沿个数、高电平时间、低电平时间。 ②测量两路信号异或后的信号的高电平时间。③测量一个周期内的高电平时间和低电平时间。

数据处理模块：以频率数据处理模块为例，首先凭借检测模块给出的原始数据计算出频率的值，然后通过加三移位法把二进制的频率转换成BCD码，然后对BCD码进行编码转换成LCD1602需要的数据。其余俩数据处理模块基本大同小异。

LCD1602：用简单的状态机就可以驱动。

硬件方面

题目给的电压都在1V以下，甚至有低于100mV的情况，这种电压不经处理的话显然FPGA是无法识别的，因此我们需要做两件事，第一件事是把电压放大至一个稳定的大小，第二件事是把正弦波整型成方波。需要用到AGC电路和TLV3501一类的电路，这部分网上资料很多，在这里我就不多说了。不过硬件方面确实要多花些时间，比起FPGA硬件对精度的影响要大的多。
